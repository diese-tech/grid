# UI Component Language

## Purpose

This doc defines how interface pieces should behave and feel inside The Grid. Components should feel like parts of a living environment instead of generic web widgets.

---

# Q/A Summary

1. Components should adapt to where they exist in The Grid.

2. Lab components can feel like specimen fact sheets, research panels, or operational modules.

3. Construction and rough zones can feel more like tech-tree nodes, floating screens, or unfinished systems.

4. Buttons should feel like a mix between game-menu buttons and tactical UI controls.

5. Project nodes should contain status, purpose, current state, screenshots, links, lessons, latest activity, and future plans.

6. Panels should stay structurally stable while their content changes contextually.

---

# Component Philosophy

The Grid should not use generic SaaS cards.

Components should feel:

- contextual
- spatial
- operational
- modular
- atmospheric
- authored

Different sectors can use different component moods while still feeling connected.

---

# Panel Behavior

The environment should remain stable while information updates contextually.

Avoid:

- endless floating windows
- chaotic draggable desktop simulation
- fake operating-system clutter

Preferred behavior:

- stable anchored layout
- nested information states
- expanding focus panels
- contextual content replacement
- sector-aware transitions

---

# Node Components

Nodes are one of the core identity pieces of The Grid.

Node inspiration:

- RPG skill trees
- Civ 6 technology trees
- operational maps
- connected research systems

Nodes should feel:

- meaningful
- interconnected
- evolving
- explorable

---

# Project Node Information

Project nodes can include:

- operational state
- screenshots
- latest activity
- purpose
- origin story
- future plans
- linked systems
- lessons learned
- experimental status
- connected projects

Projects should feel alive instead of archived.

---

# Buttons

Buttons should feel:

- tactile
- responsive
- intentional
- lightly game-inspired
- operational

Avoid:

- flat generic buttons
- crypto-dashboard controls
- hyper-futuristic gimmicks
- exaggerated hover effects

---

# Card Variants

## Operational Panels

Used for:

- dashboards
- telemetry
- active systems
- project status

Should feel:

- clean
- layered
- structured
- restrained

## Lab Specimen Panels

Used for:

- experiments
- archived concepts
- weird systems
- raw ideas

Should feel:

- slightly rougher
- more personal
- more layered
- more human

## Construction / Raw Zones

Used for:

- unfinished systems
- inactive sectors
- archived fragments
- work-in-progress pages

Can feel:

- dimmer
- more liminal
- partially incomplete
- atmospheric

---

# Interaction Rules

Components should react:

- smoothly
- contextually
- intentionally

Interaction should reward curiosity.

Avoid making the interface feel like a toy.

---

# Component Guardrail

If the UI could exist inside a generic dashboard template pack, it is wrong.

The Grid's components should feel tied to:

- environment
- atmosphere
- system state
- emotional context
